# RepositorySpec Specification

**Status:** Draft v0.1

## Purpose

`RepositorySpec` defines the conventions and structure of a repository that
supports the Agentic Engineering system.  While ProjectSpec describes the
development environment and GameSpec describes the game design, RepositorySpec
describes how the repository should be organised and what ancillary files are
required for the runtime to function.  A well‑defined RepositorySpec allows
the cockpit and runtime to bootstrap a new project with predictable
behaviour, integrate with GitHub, and manage state consistently.

## Scope

This specification covers directory layout, GitHub configuration, label and
branch conventions, and other repository‑level metadata.  It does not define
the build commands (see ProjectSpec) nor the design of the game (see
GameSpec).  It also does not specify how workflows and tasks are executed;
that is covered by AgentSpec and RuntimeSpec.

## Design Goals

1. **Consistency.** All projects using the Agentic Engineering system should
   share a common structure so that tools can work uniformly.
2. **Transparency.** It should be clear where specifications, prompts,
   documents and artefacts live within the repository.
3. **GitHub integration.** The repository should leverage GitHub features
   effectively: labels, milestones, templates and Actions.
4. **Human friendliness.** Developers should be able to navigate the
   repository without special tooling.  The directory names and file names
   should be intuitive.
5. **Extensibility.** The layout should accommodate additional specifications
   or artefacts without breaking older versions of the runtime.

## Directory and File Layout

At the top level of the repository, the following directories and files
**SHOULD** exist:

```
/
├── .agent/              # Agentic configuration and runtime data
│   ├── workflows/       # YAML definitions of missions and workflows
│   ├── prompts/         # Prompt library files (.md or .prompt)
│   ├── models.yaml      # Model router configuration
│   ├── labels.yaml      # GitHub label definitions (optional)
│   ├── context/         # Generated context files (repo index, summaries)
│   └── state/           # Transient runtime state (ignored by git)
├── .gamespec/           # Machine‑readable game design files
│   └── phoenix.gamespec.yaml  # Primary GameSpec YAML
├── docs/                # Human‑readable documentation
│   ├── prds/            # Product Requirements Documents (PRDs)
│   ├── adrs/            # Architectural Decision Records (ADRs)
│   ├── architecture/    # Additional architecture notes
│   └── wiki/            # Generated Wiki pages (optional)
├── .github/             # GitHub configuration
│   ├── ISSUE_TEMPLATE/  # Issue templates for features, bugs, questions
│   ├── PULL_REQUEST_TEMPLATE.md
│   ├── workflows/       # CI/CD workflows
│   └── labels.yml       # Optional label sync config
├── projectspec.yaml     # Project specification (see ProjectSpec)
├── README.md            # High‑level description of the project
└── <source tree>        # Code, assets and other content
```

### `.agent/`

The `.agent/` directory contains configuration and runtime support files:

* `workflows/` – YAML files describing missions and workflows.  Each file may
  contain one or more mission or workflow definitions following the
  AgentSpec grammar.  Filenames should be kebab‑case describing the
  workflow (e.g. `feature-development.yaml`).
* `prompts/` – The prompt library.  Each prompt is stored as a separate
  Markdown (`.md`) or text (`.prompt`) file.  See Prompt Library
  Specification for details.
* `models.yaml` – Configuration for the model router, listing available
  models, their providers, context windows, safe prompt budgets and
  preferences.  This file is read by the runtime when selecting models.
* `labels.yaml` – Optional file mapping human‑readable labels to GitHub
  labels and describing the state machine used by AgentSpec workflows.  If
  present, the runtime may use it to initialise or validate labels in the
  repository.
* `context/` – Auto‑generated context summaries (repo map, dependency map,
  index of ADRs, etc.).  These files are regenerated by maintenance
  workflows and MUST NOT be committed to the repository.
* `state/` – Transient runtime state, such as worktree metadata, model
  tokens used and agent run logs.  This directory SHOULD be ignored by git
  (`.gitignore`).

### `.gamespec/`

This directory contains the primary GameSpec YAML file and any
supporting files.  The filename MUST end with `.gamespec.yaml`.  Only one
top‑level GameSpec file is allowed; however subgames or expansions may be
included via the `includes` field in GameSpec.

### `docs/`

* `prds/` – PRDs generated by PRD workflows.  Each file name should match
  the issue or design number (e.g. `004-radar-contacts.md`).  Agents
  update these files when refining feature specifications.
* `adrs/` – Architectural Decision Records documenting key technical
  decisions.  The runtime may reference these to resolve ambiguities.
* `architecture/` – High‑level architecture documents, diagrams and
  narratives written by human engineers.  Agents may summarise or extract
  information from these files.
* `wiki/` – If the project uses a generated GitHub Wiki, this directory may
  store local copies of the wiki pages or templates used to generate them.
  The runtime may update the wiki via the GitHub API.

### `.github/`

This folder contains configuration files for GitHub features:

* `ISSUE_TEMPLATE/` – Markdown templates for new issues.  At minimum,
  templates for feature requests and bug reports should exist.  Agents
  populate these when creating new issues on behalf of users.
* `PULL_REQUEST_TEMPLATE.md` – A template for pull requests that includes
  checklists for tests, documentation and manual testing.  Agents may prefill
  this template when opening PRs.
* `workflows/` – GitHub Actions workflows for CI/CD.  These SHOULD run
  `build` and `test` commands defined in ProjectSpec, enforce style checks
  and, optionally, generate GameSpec or context summaries.
* `labels.yml` – Optional configuration for label synchronisation via
  [probot/label-sync](https://github.com/probot/label-sync) or similar
  tools.  It defines the canonical set of labels (e.g. `type:feature`,
  `state:needs-grilling`, etc.).

### Project Specification File

`projectspec.yaml` must reside in the repository root and conform to
ProjectSpec.  The runtime reads this file to know how to build and test the
project.

## Label Conventions

RepositorySpec standardises the GitHub labels used to track issue and
workflow state.  The runtime relies on these labels to determine the
progress of features and bugs.  A typical set of labels includes:

| Label                   | Meaning |
|-------------------------|---------|
| `type:feature`          | A feature request |
| `type:bug`              | A bug report |
| `state:needs-grilling`  | Waiting for clarifying questions to be answered |
| `state:prd-draft`       | PRD is being drafted |
| `state:needs-human`     | Paused awaiting human input |
| `state:ready-for-work`  | Ready for implementation tasks |
| `state:agent-working`   | An agent is currently processing this issue |
| `state:blocked`         | Cannot proceed automatically (e.g. context overflow) |
| `state:pr-open`         | Implementation PR has been opened |
| `state:ci-failed`       | Tests or linting failed in CI |
| `state:human-review`    | Awaiting human code review |
| `state:done`            | Completed and merged |

Projects may extend this set but SHOULD document any additions in
`labels.yaml`.

## Branch Conventions

Agents MUST NOT commit directly to the default branch (typically `main` or
`master`).  Instead, they create branches following the pattern:

```
agent/<issue-number>-<slug>
```

where `<issue-number>` is the GitHub issue number and `<slug>` is a short
description (lowercase, hyphen separated).  For example: `agent/42-radar-
contact-classification`.

Branches created for manual work (e.g. prototypes or design drafts) SHOULD
follow a similar pattern but use a different prefix (e.g. `design/` or
`experiment/`).

## Issue and Pull Request Templates

Template files in `.github/ISSUE_TEMPLATE/` guide users in filing new
feature requests, bug reports or questions.  Templates SHOULD include fields
for: title, description, expected behaviour, actual behaviour, reproduction
steps and acceptance criteria.  Agents populate these fields when
automatically creating issues from GameSpec questions or when decomposing
design tasks.

The pull request template (`PULL_REQUEST_TEMPLATE.md`) SHOULD include a
checklist: ensuring tests pass, verifying formatting, linking the related
issue, including manual test notes and updating documentation.  Agents use
this template to generate PR descriptions automatically.

## GitHub Actions and CI/CD

Repositories must define CI workflows in `.github/workflows/` that run the
build and test commands defined in ProjectSpec.  These workflows SHOULD
report status checks back to the relevant pull requests so that the runtime
and cockpit can display CI results.  Additional workflows may run linting
tools, build WebAssembly bundles, package releases or deploy documentation.

## Integration with Other Specs

* **AgentSpec** interacts with RepositorySpec by using the directory layout to
  locate workflow definitions, prompts and model configurations.  It uses
  branch and label conventions when creating or updating issues and PRs.
* **GameSpec** files reside in `.gamespec/`; RepositorySpec ensures that
  agents can find them easily.  Wiki generation workflows defined in
  AgentSpec update the `docs/wiki/` directory.
* **ProjectSpec** lives at the repository root.  RepositorySpec requires its
  presence and may use it to validate CI workflows or show warnings in the
  cockpit if the build or test definitions are missing.

## Future Extensions

Future versions of RepositorySpec may define conventions for monorepos,
third‑party dependencies, containerisation (e.g. Dockerfiles) or packaging
artefacts.  They may also standardise integration with other platforms like
Discord, Slack or issue trackers beyond GitHub.
